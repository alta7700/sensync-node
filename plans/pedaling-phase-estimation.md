# План: грубая оценка фазы педалирования и окон активности по IMU + EMG

Дата исследования: 14 марта 2026 года.

## Цель

Понять, как в `sensync2` реализовать отдельный runtime-обработчик, который по `IMU` и `EMG` с датчика на `vastus lateralis` будет оценивать:

- грубую фазу педалирования;
- окна мышечной активности;
- опционально четверти цикла, если качество сигнала позволит.

Сразу фиксируем границу задачи: речь не про восстановление точного угла педали `0..360°`, а про грубую фазовую разметку цикла с допустимой ошибкой порядка `10-20%` длительности оборота.

## Что считаем исходными допущениями

На текущем этапе считаем верными такие допущения:

- на датчике есть как минимум один `IMU`-канал, пригодный для отслеживания периодики движения ноги;
- тот же датчик или соседний канал даёт `EMG` с `vastus lateralis`;
- положение датчика между сессиями похоже, но не идеально воспроизводимо;
- cadence во время рабочих интервалов относительно стабилен;
- интерфейс `Lode Excalibur Sport` не даёт нам публично доступного точного угла педали, поэтому алгоритм нельзя строить вокруг абсолютного угла педали.

Практический вывод из этих допущений:

- для `2` фаз задача выглядит реалистично;
- для `4` фаз это должна быть производная дискретизация нормированного цикла, а не обещание истинных биомеханических опорных событий;
- `EMG` нужен не как замена фазе, а как корректирующий канал для `onset/offset` внутри грубой фазы.

## Что считаем успехом

Первый практичный критерий успеха:

- потоковый обработчик устойчиво делит цикл хотя бы на `2` фазы;
- обработчик выдаёт интервалы активности `vastus lateralis`, уточнённые по `EMG`;
- ошибка разметки остаётся в пределах примерно `10-20%` цикла на ровном педалировании;
- деградация качества выражается явно через оценку уверенности или деградированный режим, а не через молчаливую выдачу случайной фазы.

Что не считаем целью первого этапа:

- абсолютный угол педали;
- реконструкцию полной биомеханики цикла;
- точную привязку к `top dead center` / `bottom dead center` без внешнего референса на кривошипе;
- работу без калибровки в произвольных режимах, включая сильные рывки cadence и резкие изменения посадки.

## Архитектурное размещение

Эту логику нельзя прятать ни в транспортный адаптер, ни в `ui-gateway`.

Рекомендация:

- сделать отдельный worker-плагин в `packages/plugins-*`, рабочее имя `pedaling-phase-detector`;
- подписывать его на уже существующие `signal.batch` потоки от `IMU`, `EMG` и, при наличии, от внешнего cadence-источника;
- публиковать наружу тоже только через `signal.batch`, не вводя новый тип shared event без крайней необходимости.

Почему это решение совместимо с текущей моделью `runtime -> plugins -> ui-gateway -> client-runtime`:

- transport-слой должен только доставлять сырые данные устройства;
- оценка фазы и активности уже является вычисляемой доменной логикой;
- `signal.batch` в текущих контрактах уже поддерживает `label-batch`, что подходит для публикации состояний и смены фаз;
- `ui-gateway` должен только materialize'ить результат, а не считать фазу в renderer.

Опорные места в репозитории:

- [packages/core/CONTRACTS.md](../packages/core/CONTRACTS.md) — временная модель и семантика `signal.batch`;
- [packages/core/src/events.ts](../packages/core/src/events.ts) — `frameKind = label-batch`;
- [packages/plugins-fake/src/interval-label-adapter.ts](../packages/plugins-fake/src/interval-label-adapter.ts) — минимальный пример state-like label потока;
- [apps/runtime/README.md](../apps/runtime/README.md) — требования к plugin worker и data-path.

## Предлагаемые входы обработчика

Минимальный вход:

- один канал `IMU`, предпочтительно `gyro`, если он есть;
- один канал `EMG` для `vastus lateralis`.

Желательный, но не обязательный вход:

- cadence от `Lode`, `ANT+` или другого внешнего источника, если его можно читать надёжно и в session-time.

Жёсткая рекомендация по приоритету `IMU`:

1. одна ось `gyro`;
2. одна ось `acc`, если `gyro` нет;
3. комбинированная схема `gyro + acc`, если оба доступны.

Причина простая: для циклического движения ноги угловая скорость обычно даёт более чистую фазовую структуру, чем линейное ускорение, где сильнее смешиваются гравитация, вибрации и артефакты крепления датчика на коже.

## Предлагаемые выходы обработчика

Для первого этапа не нужно раздувать контракт. Достаточно ограничиться такими потоками:

- `pedaling.phase.coarse`
  - `label-batch`
  - значения `0 | 1`
  - смысл: две грубые половины цикла;
- `pedaling.activity.vastus-lateralis`
  - `label-batch`
  - значения `0 | 1`
  - смысл: окно активности мышцы после коррекции по `EMG`.

Опциональные диагностические потоки второго этапа:

- `pedaling.phase.quadrant`
  - `label-batch`
  - значения `0..3`;
- `pedaling.phase.confidence`
  - `uniform-signal-batch` или `irregular-signal-batch`
  - `f32`
  - значение `0..1`;
- `pedaling.cycle.period-ms`
  - низкочастотный служебный поток для отладки стабильности цикла.

Важное ограничение:

- `quadrant` надо трактовать именно как четверти нормированного цикла;
- пока нет внешнего энкодера, нельзя называть эти значения истинным углом педали.

## Теоретическая схема алгоритма

### 1. Захват и выравнивание по времени

Обработчик живёт в session-time и работает поверх уже нормализованных `signal.batch`.

Правила:

- не использовать wall-clock как источник истины;
- буферизовать короткое окно последних данных по `IMU` и `EMG`;
- при разной частоте дискретизации не пытаться насильно свести всё к одному сырому sample rate, если это не нужно для принятия решения;
- для потоковой логики допускать небольшую внутреннюю задержку окна, если это заметно улучшает устойчивость детекции.

Практически здесь нужен кольцевой буфер на несколько последних секунд.

### 2. Предобработка `IMU`

Если есть `gyro`, он становится главным фазовым каналом. Если есть только `acc`, алгоритм остаётся тем же по форме, но с более консервативными порогами.

Минимальная обработка:

- убрать медленный дрейф и DC-компоненту;
- применить адаптивную полосовую фильтрацию вокруг ожидаемой частоты педалирования;
- оценить текущий период цикла по автокорреляции, спектральному пику или другой устойчивой оценке основной частоты;
- подавить ложные срабатывания через защитное окно, связанное с ожидаемой длительностью полуцикла.

Почему нужен адаптивный полосовой фильтр:

- фиксированный фильтр быстро ломается при смене cadence;
- вам нужна не идеальная форма волны, а устойчивая периодика и повторяемые точки цикла.

### 3. Выбор якорного события цикла

Обработчику нужно повторяемое якорное событие, которое делит соседние циклы.

Рабочие варианты:

- локальный максимум;
- локальный минимум;
- переход через ноль с правильным направлением;
- пик после гистерезиса.

Выбор якорного события надо делать не "по красоте графика", а по воспроизводимости на записи. Важнее стабильный однотипный маркер, чем его красивая биомеханическая интерпретация.

Итог этой стадии:

- есть последовательность якорных времен `t_cycle[n]`;
- между двумя соседними якорями строится нормированная фаза `0..1`.

### 4. Грубая фазовая разметка

Базовый режим:

- первая половина между `0.0` и `0.5` даёт `pedaling.phase.coarse = 0`;
- вторая половина между `0.5` и `1.0` даёт `pedaling.phase.coarse = 1`.

Режим `4` фаз:

- цикл режется на четверти `0.0-0.25`, `0.25-0.5`, `0.5-0.75`, `0.75-1.0`;
- эти четверти трактуются как удобная дискретизация цикла, а не как доказанные биомеханические события.

Из этого следует важное проектное решение:

- `2` фазы должны быть основным поддерживаемым режимом;
- `4` фазы надо строить поверх уже устойчивой нормированной фазы, а не отдельной эвристикой.

### 5. Предобработка `EMG`

Если доступен сырой `EMG`, нужен типовой конвейер:

- полосовой фильтр в рабочем EMG-диапазоне;
- выпрямление;
- огибающая через low-pass или скользящее RMS-окно;
- адаптивная оценка базового уровня и порога по окну последних секунд.

Если датчик отдаёт уже огибающую, часть конвейера пропускается, но это надо фиксировать в конфиге, а не угадывать внутри обработчика.

Цель этой стадии:

- получить не "красивый EMG-график", а бинарное или квазибинарное представление вероятности активности мышцы.

### 6. Коррекция фазы по `EMG`

Здесь не надо делать ошибку и пытаться вывести весь цикл только из `EMG`.

Правильная схема:

- `IMU` задаёт грубое положение внутри цикла;
- `EMG` уточняет начало и конец активного окна внутри ожидаемой половины или четверти цикла.

Практически это означает:

- искать `EMG onset` только в допустимом окне, где активность мышцы вообще ожидается;
- искать `EMG offset` тоже в ограниченном окне;
- если `EMG` внезапно активен вне ожидаемой зоны, не перестраивать всю фазу мгновенно, а понижать оценку уверенности и ждать подтверждения на нескольких циклах.

Это лучше, чем полностью отдавать фазу на откуп `EMG`, потому что `EMG` чувствителен к шуму, артефактам контакта и межсессионным отличиям крепления.

### 7. Устойчивость и подавление дрожания состояний

Без этой стадии обработчик будет дёргаться.

Нужны как минимум:

- гистерезис для переходов;
- минимальная длительность состояния;
- запрет на слишком частые переключения внутри одного предполагаемого цикла;
- оценка уверенности, которая падает при пропуске якорей, сильном изменении периода или противоречии `IMU` и `EMG`.

Если оценка уверенности падает ниже порога, разумнее:

- перейти в деградированный режим;
- продолжать выдавать только `2` фазы;
- либо временно заморозить уточнение окна активности, но не выдавать ложную точность.

## Предлагаемая машина состояний обработчика

Теоретически обработчик удобно держать в четырёх режимах:

1. `warming-up`
   - накопление данных и первичная оценка периода;
2. `tracking`
   - обычная потоковая работа с устойчивой фазой;
3. `degraded`
   - цикл ещё угадывается, но оценка уверенности низкая, поэтому отключается часть уточнений;
4. `lost`
   - периодика пропала, активная разметка фаз временно не публикуется или публикуется как неизвестное или замороженное состояние.

Переходы между режимами надо делать только по окнам и счётчикам подтверждений, а не по одиночному шумному батчу.

## Почему схема `2 фазы + коррекция по EMG` выглядит сильнее, чем прямые `4 фазы`

Потому что для вашей постановки задачи основная ценность — отделить интервалы активности, а не красиво назвать каждую четверть оборота.

С инженерной точки зрения:

- `2` фазы требуют меньше предположений о точной геометрии движения;
- их проще стабилизировать на одном `IMU`-канале;
- `EMG` естественно уточняет активное окно внутри одной грубой фазы;
- `4` фазы можно добавить позже почти бесплатно, если уже есть надёжная нормированная фаза.

Если пытаться начать сразу с `4` фаз, есть риск получить более красивую, но менее честную систему.

## Ограничения и риски

Главные ограничения:

- один канал `IMU` на `vastus lateralis` не является прямым датчиком угла кривошипа;
- форма сигнала зависит от посадки, усилия, cadence и качества наклейки датчика;
- `acc`-канал особенно чувствителен к паразитным ускорениям и артефактам движения кожи;
- `EMG` может плыть по базовому уровню, контакту и движению кабеля или корпуса датчика.

Из этого следуют риски:

- привязка фазы между сессиями будет хуже, чем внутри одной сессии;
- при резком изменении cadence или техники педалирования якорное событие может смещаться;
- без внешнего референса нельзя честно обещать совпадение с физическими верхней и нижней мёртвыми точками.

Поэтому первый этап должен быть ориентирован на устойчивую внутри-сессии сегментацию, а не на абсолютную межсессионную геометрию.

## Предлагаемый MVP

Если резать задачу прагматично, первый MVP должен делать только это:

1. принимать `IMU` и `EMG` потоки;
2. находить периодический маркер цикла по `IMU`;
3. делить цикл на `2` половины;
4. по огибающей `EMG` уточнять начало и конец окна активности;
5. эмитить:
   - `pedaling.phase.coarse`;
   - `pedaling.activity.vastus-lateralis`;
   - опционально `pedaling.phase.confidence` для отладки.

Что не надо пихать в MVP:

- `4` фазы как обязательный продуктовый контракт;
- сложную автоматическую логику выбора оси, если ось реально одна;
- агрессивную попытку "переучивать" фазу на каждом батче;
- UI-эвристику вместо runtime-сигнала.

## План реализации после утверждения теории

1. Зафиксировать входные каналы и их частоту дискретизации.
2. Описать внутренний конфиг обработчика:
   - размер окна;
   - тип `IMU`-канала;
   - фильтры;
   - пороги оценки уверенности;
   - режим `2-phase only` или `2-phase + quadrants`.
3. Сначала собрать автономный анализатор на записанных окнах, чтобы сравнивать детекцию якорей и `EMG onset/offset`.
4. После этого вынести ту же логику в потоковый worker-плагин.
5. Сразу заложить диагностические выходы, чтобы не дебажить фазу вслепую.
6. Только после нескольких живых сессий решать, нужен ли вообще публичный `quadrant`-канал.

Здесь важно не срезать путь. Если сразу писать потоковый плагин без автономной проверки на реальных записях, вы рискуете перепутать ошибки алгоритма с ошибками интеграции потока.

## Что нужно уточнить перед кодом

Есть два вопроса, которые уже влияют на конкретную реализацию:

1. У вас реально доступен `gyro`, или на первом этапе надо исходить из одной оси `acc`?
2. Первый проход нужен сразу как потоковый runtime plugin, или сначала допустим автономный анализатор по уже записанным данным?

Пока эти ответы не получены, самый безопасный рабочий сценарий такой:

- проектировать алгоритм под потоковый runtime;
- считать, что есть один основной `IMU`-канал;
- делать основную ставку на `2` фазы и `EMG`-коррекцию активности.
