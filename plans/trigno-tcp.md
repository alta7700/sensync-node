# План: поддержка Delsys Trigno по TCP

Дата исследования: 10 марта 2026 года.

## Цель

Понять, можно ли в `sensync2` поддержать `Trigno` без возврата к старому Python runtime и есть ли для этого зрелая Node/Electron-библиотека.

## Что уже делает текущий `sensync`

Проверено по `../sensync/drivers/trigno.py`.

Текущий драйвер:
- открывает отдельный command socket;
- открывает один или несколько data sockets;
- шлёт ASCII-команды, завершённые `\\r\\n\\r\\n`;
- ждёт `OK` в ответе;
- читает бинарные `float32` данные;
- собирает неполные пакеты через аккумулятор;
- раскладывает данные по каналам.

Из этого следует важный вывод: у нас уже нет признаков, что тут нужен тяжёлый vendor SDK внутри runtime. Транспортная часть выглядит как обычный TCP-клиент с несколькими сокетами.

## Готовые библиотеки

### Node/TypeScript библиотека под Trigno

На 10 марта 2026 года удовлетворяющего Node-решения я не нашёл.

Что проверял:
- поиск репозиториев на GitHub по `trigno node`, `delsys trigno node`, `delsys emg node`;
- поиск npm-пакетов по `trigno`, `delsys`, `delsys-trigno`.

Результат:
- заметного активного и популярного Node-пакета нет;
- готового TypeScript runtime-слоя под Trigno фактически нет.

Это не мелочь. Здесь лучше не придумывать себе "скрытый рынок библиотек": если бы зрелое решение существовало, оно было бы заметно хотя бы по npm и GitHub.

### Запасной ориентир: Python `pytrigno`

- Репозиторий: [axopy/pytrigno](https://github.com/axopy/pytrigno)
- Активность:
  - `42` stars
  - последний push: `2019-05-25`
- Вывод:
  - это не решение для `sensync2` на Node;
  - но это полезный резервный источник для сверки поведения и командного протокола, если понадобятся дополнительные детали.

## Оценка самостоятельной реализации

Здесь ситуация лучше, чем кажется.

Если опираться на уже существующий драйвер в `sensync`, то самостоятельная реализация в `sensync2` выглядит так:
- `net.Socket` для command channel;
- `net.Socket` для data channels;
- маленький state machine для `initialize/start/stop/close`;
- бинарный парсер `float32`;
- аккумулятор неполных пакетов;
- адаптерный plugin, который эмитит `signal.batch`.

### Сложность

- базовая реализация под известную модель устройства: низкая или средняя;
- универсальная реализация под разные ревизии и режимы Trigno: средняя;
- реализация без доступа к железу: рискованная, потому что часть поведения придётся угадывать по обратной инженерии текущего драйвера.

### Где можно ошибиться

- последовательность команд и жизненный цикл сокетов;
- режим `BACKWARDS COMPATIBILITY`;
- разница между моделями или версиями Trigno;
- точное соответствие количества каналов, портов и размера кадра;
- обработка reconnect и неполных TCP-пакетов.

Но важно зафиксировать: это не тот случай, где нужно реализовывать огромную внешнюю спецификацию вроде BLE stack или сам формат HDF5. Здесь область задачи ограничена и уже partially decoded текущим Python-кодом.

## Рекомендация для `sensync2`

Рекомендую не искать дальше "магическую" Node-библиотеку и делать собственный `trigno-tcp-adapter` plugin.

Практичный план:

1. Вынести из текущего Python-драйвера фактический контракт:
   - команды;
   - порты;
   - размер кадра;
   - число каналов на порт;
   - режимы запуска и остановки.
2. Реализовать чистый Node plugin на `net.Socket`.
3. Сначала поддержать только тот сценарий, который уже проверен текущим `sensync`.
4. Добавить запись сырого traffic/logging для первых прогонов на железе.
5. Только если всплывёт vendor-зависимая нестабильность, рассматривать sidecar на Python.

Итоговая оценка:
- поддержка Trigno в `sensync2` реалистична;
- готовой хорошей Node-библиотеки нет;
- собственная реализация оправдана;
- риск проекта: средний, но управляемый при наличии устройства для тестирования.
